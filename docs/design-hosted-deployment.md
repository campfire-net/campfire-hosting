## 1. Architecture Overview

Two deployment tracks, one binary, one image, one cloud.

### Track 1: Hosted Service (mcp.getcampfire.dev)

Free, multi-tenant, ephemeral. The adoption on-ramp.

- **Compute:** Azure Container Apps (Consumption plan). Scale-to-zero when idle.
- **Store:** In-memory cache (hot path) + Azure Table Storage (durable journal). Container holds live state in RAM; Tables are the write-through persistence layer.
- **Delivery:** SSE-first. Agents hold persistent SSE connections. Message delivery is in-memory fan-out with zero table reads. Polling is a fallback for clients that cannot hold SSE connections.
- **Cost:** ~$0 at low usage (within ACA free grant), ~$3/mo with persistent SSE connections (see Section 10).
- Zero-friction: operator signs up, gets API key, agents connect via MCP immediately.
- Multi-tenant with per-operator isolation. Agents belong to operators; sessions are token-scoped table partitions.
- Hard agent cap: ~1,000 agents total across all operators. Beyond that, graduate to managed app or self-host.
- Ephemeral sessions: 48-hour inactivity TTL. `campfire_export` is the persistence valve.
- No SLA, no guaranteed uptime. This is distribution infrastructure, not a revenue platform.

### Track 2: Azure Managed Application (Marketplace)

Paid, single-tenant, persistent. The enterprise revenue channel.

- **Compute:** Azure App Service B1 (Linux, container). Always warm, zero cold start.
- **Store:** Azure Table Storage in the customer's storage account.
- **Cost to customer:** ~$17-20/mo infrastructure + license fee ($0-$199/mo depending on tier).
- Customer deploys cf-mcp into THEIR Azure subscription via Azure Marketplace one-click.
- Operator (campfire-net) manages remotely via managed resource group with Contributor access.
- Single-tenant: customer's agents, customer's keys, customer's telemetry.
- Data sovereignty: everything stays in the customer's subscription and region.
- SLA backed by Azure infrastructure (99.9% for App Service).

### Interoperability

Both tracks run the same binary, speak the same protocol, and interoperate natively via HTTP transport. An agent on the hosted service can coordinate with an agent on a managed app instance in the same campfire. The protocol does not distinguish deployment modes.

```
┌─────────────────────────────────────────────────────────┐
│ Hosted: mcp.getcampfire.dev (ACA Consumption)           │
│ Multi-tenant, ephemeral, scale-to-zero                  │
│ Purpose: adoption, experimentation, small-scale use     │
├─────────────────────────────────────────────────────────┤
│ Managed App: Azure Marketplace (App Service B1)         │
│ Single-tenant, persistent, customer's subscription      │
│ Purpose: enterprise revenue, data sovereignty, SLA      │
└─────────────────────────────────────────────────────────┘
         ▲                                      ▲
         │     Both are HTTP transport peers     │
         └──────── campfire mesh ────────────────┘
```

## 2. Operator Identity

Two identity layers. One for billing, one for protocol participation.

### Layer 1: Operator

The operator is the billing entity -- the human or organization that signs up, pays for throughput, and manages agents. An operator has:

- **Operator ID**: UUID, assigned at sign-up.
- **API key**: Secret key for authenticating API calls and creating agents. Operators can rotate keys. Multiple keys supported (e.g., production vs. staging).
- **Billing profile**: Payment method, usage tier, invoice history.
- **Agent quota**: Maximum concurrent agents (hard cap per tier -- see Section 2a).

Sign-up flow:
1. Operator registers at getcampfire.dev (email + payment method for paid tiers, email-only for free tier).
2. System generates operator ID + initial API key.
3. Operator uses API key to create agents via `campfire_init`.

### Layer 2: Agent

The agent is the campfire participant -- the Ed25519 keypair that signs messages, joins campfires, and coordinates with other agents. An agent belongs to exactly one operator.

- **Agent identity**: Ed25519 keypair (generated by `campfire_init`).
- **Session token**: Scopes the agent's state in Azure Tables.
- **Operator binding**: Every agent is created under an operator account.

### Auth Flow

```
Operator API Key
  → POST /mcp (Authorization: Bearer <api-key>)
    → campfire_init
      → Validate API key → resolve operator_id
      → Check agent quota (reject if at cap)
      → Generate Ed25519 keypair
      → Create session: store agent under operator_id
      → Return session token + public key
    → Subsequent calls
      → Authorization: Bearer <session-token>
      → Session token scopes all state access
```

The API key is used exactly once per agent lifecycle: at `campfire_init`. After that, the session token authenticates all requests. This keeps the API key out of long-lived agent sessions.

### Per-Operator Metering

All usage is metered at the operator level:

- **Active agents**: count of non-expired sessions under the operator.
- **Messages sent**: total `campfire_send` calls across all agents.
- **Storage**: total entity count across all agent sessions.

The `/health` endpoint reports aggregate metrics. Per-operator metrics are queryable via the operator dashboard (Phase 2).

### Managed App: Simplified Identity

For managed app deployments, the operator IS the customer. Single-tenant means:

- The API key layer MAY be unnecessary -- the customer controls network access to the App Service.
- If the customer wants API key auth (e.g., multiple teams sharing one instance), the same two-layer model applies.
- Agent quota is unlimited (customer pays their own compute).

## 2a. Business Model

### Hosted Service (mcp.getcampfire.dev)

The hosted service is a free/cheap adoption on-ramp with a hard agent cap. It is NOT a revenue-generating platform at scale.

| Tier | Monthly Cost | Agent Cap | Message Rate | Support |
|------|-------------|-----------|--------------|---------|
| Free | $0 | 50 agents | 5 msgs/hr/agent | Community |
| Starter | $9/mo | 200 agents | 30 msgs/hr/agent | Email |
| Growth | $29/mo | 1,000 agents | 60 msgs/hr/agent | Email, 8h |

**Hard cap: ~1,000 agents on the hosted service.** Beyond that, operators graduate to a managed app deployment or self-host. The hosted service is distribution infrastructure, not a scalable platform.

Revenue from the hosted service covers its own costs (storage, compute, domain) and nothing more. It exists to reduce friction for adoption.

### Managed App (Azure Marketplace)

The managed app is the revenue channel. Operators who outgrow the hosted service deploy cf-mcp into their own Azure subscription.

| Tier | Monthly License | Agent Capacity | SLA | Support |
|------|----------------|----------------|-----|---------|
| Community | Free | Up to 10 agents | None | Community |
| Team | $49/mo | Up to 100 agents | 99.9% | Email, 24h |
| Enterprise | $199/mo + $0.50/100 additional | Unlimited | 99.9% | Priority, 4h |

Azure Marketplace commission: 3%. Net revenue per Team customer: $47.53/mo at zero compute cost to the operator (campfire-net).

The managed app is single-tenant: the customer's agents, the customer's keys, the customer's Azure subscription. The operator (campfire-net) manages remotely via managed resource group access.

### Self-Hosted

Operators with existing infrastructure or specific compliance requirements run cf-mcp themselves. No license fee. Uses SQLite backend (no Azure dependency). Support is community-only.

### Graduation Path

```
Free hosted (50 agents) → Starter (200) → Growth (1,000)
                                              ↓
                              Managed App (Team/Enterprise)
                                     or
                              Self-Hosted (no cap, no fee)
```

The hosted service deliberately caps before costs matter. The 100K/1M agent scenarios in the cost architecture (Section 10) are guidance for managed app customers sizing their own deployments, not hosted service projections.

## 3. Azure Table Storage Schema

Azure Table Storage is the store backend for both tracks. This eliminates the entire SQLite-on-Azure problem: no WAL mode, no disk, no volume mount. The container is stateless. Scale-to-zero works because there is nothing on disk to lose.

### Characteristics

- Schemaless key-value store: PartitionKey + RowKey per entity.
- Sub-10ms latency for point reads and single-partition scans (<1000 entities).
- ~$0.036 per 10K transactions (standard tables, LRS).
- ~$0.045/GB-month for storage.
- Strong consistency within a partition (read-after-write guaranteed).
- Entity Group Transactions: atomic batch writes within a single partition (up to 100 entities, 4MB). Saves HTTP round-trips but NOT transaction costs -- Azure bills per entity operation, not per HTTP call.
- No cross-partition transactions.

### Table Design

All tables live in one storage account. Table names are prefixed to avoid collision with customer data.

**Table: `cfoperators`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `operator_id` | `_meta` | `email`, `created_at` (int64), `tier` (string), `agent_cap` (int), `billing_status` (string) |

**Table: `cfapikeys`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `api_key_hash` | `_meta` | `operator_id`, `created_at` (int64), `label` (string), `last_used` (int64) |

API keys are stored as SHA-256 hashes. Lookup is by hash (point read). The operator_id property links back to `cfoperators`.

**Table: `cfmessages`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `campfire_id` | `message_id` | `timestamp` (int64), `sender_pubkey`, `payload` ([]byte), `signature`, `tags` (string, comma-separated), `reply_to`, `provenance_chain` |

Queries:
- `campfire_read`: partition scan on `campfire_id`, filter `timestamp > after_ts`. Sub-10ms for partitions with <1000 messages.
- `campfire_await`: repeated partition scan with `timestamp > last_seen_ts`. Latency dominated by poll interval, not storage.
- Point read by message ID: `PartitionKey=campfire_id, RowKey=message_id`. Sub-10ms.

**Table: `cfmemberships`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `session_token` | `campfire_id` | `role`, `joined_at` (int64), `campfire_name`, `creator_pubkey` |

Queries:
- `campfire_list` (all campfires for a session): partition scan on `session_token`.
- `campfire_info` (membership for one campfire): point read.

A reverse index is NOT needed because the hosted service never queries "all members of a campfire" from the store -- membership is tracked in-protocol via campfire messages. The session only needs to know which campfires it belongs to.

**Table: `cfcursors`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `session_token` | `campfire_id` | `timestamp` (int64) |

Small table. Point reads and writes only.

**Table: `cfpeers`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `campfire_id` | `member_pubkey` | `endpoint_url`, `role`, `nonce_seed` |

Queries:
- List peers for a campfire: partition scan on `campfire_id`.
- Upsert/delete: point operations.

**Table: `cfthresholdshares`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `session_token` | `campfire_id` | `threshold`, `participant_id` (uint32), `share_data` ([]byte), `group_key` |

**Table: `cfpendingshares`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `campfire_id` | `participant_id` | `share_data` ([]byte), `claimed` (bool) |

**Table: `cfsessions`**

| PartitionKey | RowKey | Properties |
|---|---|---|
| `session_token` | `_meta` | `created_at` (int64), `last_active` (int64), `public_key`, `private_key_encrypted` ([]byte), `identity_json` ([]byte), `operator_id` |

This table replaces the on-disk `identity.json` and session directory. The session token is the partition key. Identity material is stored as entity properties. `last_active` is updated on every request for TTL-based expiry. `operator_id` links every session to its owning operator for quota enforcement and metering.

### Cross-Table Consistency

The adversary raised valid concerns: campfire_send requires writing to `cfmessages` AND updating `cfcursors` -- these are in different partitions (different partition keys) and CANNOT be batched atomically.

**Mitigation: ordered writes with idempotency.**

1. Write the message to `cfmessages` first (this is the source of truth).
2. Update `cfcursors` second.
3. If step 2 fails, the cursor is stale but the message exists. The next `campfire_read` will return the message (it queries by timestamp, not cursor). The cursor catches up on the next read.
4. Message writes use the message ID as the RowKey -- duplicate writes are idempotent (Azure Tables upsert semantics).

This is eventual consistency for cursors, strong consistency for messages. Acceptable because cursors are an optimization (skip already-read messages), not a correctness requirement.

**Operations that ARE atomically consistent:** All writes within a single campfire's messages (same partition key) can be batched in an Entity Group Transaction for atomicity. This covers the common case of "send a message and update related message metadata." Note: batching reduces HTTP round-trips (latency), not transaction costs (Azure bills per entity, not per HTTP call).

### Transaction Cost Reality

Azure Tables bills at $0.036 per 10K transactions. Each entity operation (read, write, delete) counts as one transaction, regardless of whether it is in a batch or individual HTTP call. Entity Group Transactions save HTTP round-trips (latency), not money.

With the SSE-first architecture (see Section 10), the transaction profile changes fundamentally:

- **Reads:** Near-zero table transactions. SSE fan-out delivers messages from memory. `campfire_read` hits the in-memory cache. Only cache misses (cold campfires, cold starts) touch Tables.
- **Writes:** Every message is written through to Tables for durability. Each `campfire_send` is ~2 transactions (message write + cursor update). This is the dominant cost at scale.
- **Polling:** Eliminated. `campfire_await` uses the in-memory PollBroker + SSE push. Zero table transactions for notification delivery.

The cost curve is linear with message volume, not agent count. Idle agents holding SSE connections cost zero table transactions. See Section 10 for the full cost model at scale.

## 4. Store Architecture

### Design Principle: Memory Is the Hot Path

The store architecture has three layers:

```
Layer 1: In-Memory Cache (the hot path)
  All reads served from memory. All SSE fan-out from memory.
  Per-campfire: message ring buffer (last 1000 msgs), membership set, PollBroker
  Per-agent: session metadata, campfire subscriptions, SSE channel
  Eviction: LRU by campfire when memory pressure requires it

Layer 2: Azure Table Storage (the durable journal)
  Write-through: every message persisted to Tables after in-memory delivery
  Read-on-miss: cold campfire accessed → hydrate from Tables → promote to Layer 1
  Cold start: hydrate active campfire set from Tables (see Section 12)

Layer 3: Azure Blob Storage — append blobs (Phase 3 optimization)
  Cheaper writes ($0.006/10K vs $0.036/10K for Tables — 6x cheaper)
  Cheaper reads ($0.001/10K vs $0.036/10K — 36x cheaper)
  Per-campfire append blob with daily rotation (50K append limit per blob)
  Candidate for replacing Tables as the durable journal at scale
```

This is not a traditional cache. Memory IS the primary store. Azure Tables is the durability mechanism. 99%+ of reads never touch Tables -- they hit the in-memory ring buffers. The only table reads happen on cold start (cache hydration) and cache miss (accessing a campfire that was evicted from memory).

### Memory Footprint

Per agent: ~314KB (session metadata ~2KB + message cache for ~3 campfire memberships at 100 msgs each ~300KB + goroutine overhead ~12KB).

Per campfire: ~1MB (membership roster + 1000-message ring buffer + PollBroker subscriber channels).

| Agents | Agent State | Campfire State (10 agents/campfire) | Total RAM |
|--------|-------------|-------------------------------------|-----------|
| 1,000 | ~314MB | ~100MB | ~414MB |
| 10,000 | ~3.1GB | ~1GB | ~4.1GB |
| 100,000 | ~31GB | ~10GB | ~41GB |
| 1M | ~314GB | ~100GB | ~414GB |

At 0.5 GiB per container (ACA Consumption), one container holds ~1,000 agents. At 4 GiB (ACA max), one container holds ~7,000 agents. Beyond that, partition across instances with campfire-ID-based consistent hashing.

### Current Store Interface

`pkg/store/interfaces.go` defines four sub-interfaces: `MembershipStore`, `MessageStore`, `PeerStore`, `ThresholdStore`. The concrete `*Store` (SQLite) satisfies all four. There is no unified interface and no `Close()` in the interfaces.

### What Needs to Change

Introduce a unified `Store` interface that both SQLite and Azure Tables implement, plus a `CachedStore` wrapper that adds the in-memory cache layer.

```go
// pkg/store/iface.go

// Store is the unified storage interface for campfire.
// SQLiteStore is the default for local CLI and self-hosted use.
// AzureTableStore is used in hosted and managed app deployments.
type Store interface {
    MembershipStore
    MessageStore
    PeerStore
    ThresholdStore

    // CampfireManagement
    UpdateCampfireID(oldID, newID string) error

    Close() error
}
```

The `CachedStore` wraps any `Store` backend:
- Reads: check in-memory ring buffer first, fall through to backing store on miss
- Writes: write to in-memory state, then write-through to backing store asynchronously
- SSE fan-out: triggered from in-memory write, never touches backing store

### Implementation Approach

1. Add `pkg/store/iface.go` with the unified `Store` interface (~15 LOC).
2. Rename the existing `Store` struct to `SQLiteStore`. Update all callers to use the interface type where storage is passed as a dependency.
3. Create `pkg/store/aztable/` for the Azure Table Storage backend. Implements `Store` using `github.com/Azure/azure-sdk-for-go/sdk/data/aztables`.
4. Create `pkg/store/cached/` for the in-memory cache wrapper. Implements `Store` by composing a backing `Store` with in-memory ring buffers, LRU eviction, and write-through semantics.
5. `cmd/cf-mcp/session.go` selects the backend based on configuration: `--store-backend=sqlite` (default) or `--store-backend=aztable` (when `AZURE_STORAGE_ACCOUNT` env var is set). The cached wrapper is applied automatically in hosted mode.
6. The `aztable` package is compiled unconditionally (pure Go, no CGO) but only instantiated when configured.
7. Add `pkg/store/aztable/session.go` for session identity management (replaces on-disk `identity.json`).

### Azure Tables Backend Complexity

The store interface has ~22 methods across the four sub-interfaces plus `UpdateCampfireID` and `Close`. Each method maps to 1-3 Azure Tables SDK calls. Estimated: ~400 LOC for the aztable backend, ~200 LOC for tests against Azurite emulator, ~300 LOC for the cached wrapper.

Query patterns used by campfire (point reads, range scans by partition key, upserts, deletes) are native Azure Table operations. No joins, no aggregations, no complex queries.

### What Does NOT Change

The `cf` CLI continues using SQLite (Wazero, no CGO). The protocol, transport layer, and identity management are unchanged. Only the session-scoped store backend in `cmd/cf-mcp` is affected.

## 5. ACA Deployment (Hosted Service)

### Why ACA Consumption

With a stateless container (Azure Tables handles all persistence), Azure Container Apps Consumption is the cheapest Azure compute path:

| Azure Option | Monthly (24/7) | Scale-to-Zero | Custom Domain + TLS | Container Support |
|---|---|---|---|---|
| **ACA Consumption** | **~$19/mo (0.25 vCPU, 0.5 GiB 24/7)** | **Yes (native)** | **Yes** | **Yes** |
| ACI | ~$33/mo | No (manual stop/start) | Manual | Yes |
| App Service B1 | ~$13/mo | No | Yes | Yes |
| App Service Free | $0 | Yes (20min idle) | No custom domain | No containers |
| Azure Functions | N/A | Yes | Yes | SSE incompatible |

ACA Consumption includes a free monthly grant: 180K vCPU-seconds + 360K GiB-seconds + 2M requests. For a 0.25 vCPU / 0.5 GiB container, this covers ~200 hours of active runtime and ~6.6 hours/day of continuous operation.

### Environment Variables

| Variable | Value | Required |
|---|---|---|
| `CF_EXTERNAL_URL` | `https://mcp.getcampfire.dev` | Yes -- beacons advertise this URL |
| `CF_STORE_BACKEND` | `aztable` | Yes -- selects Azure Tables backend |
| `CF_HOSTED_MODE` | `true` | Yes -- enables operator auth, agent caps, hosted-only restrictions |
| `AZURE_STORAGE_ACCOUNT` | Storage account name | Yes -- Tables backend needs this |
| `AZURE_CLIENT_ID` | (from managed identity) | Auto -- ACA injects this |

Authentication to Azure Tables uses the container's managed identity via `azidentity.NewDefaultAzureCredential()`. No storage keys in environment variables. Operator API keys are validated against the `cfapikeys` table on `campfire_init`; all subsequent requests use session tokens.

### Custom Domain and TLS

ACA supports custom domains with managed TLS certificates. DNS: CNAME `mcp.getcampfire.dev` to the ACA environment's default domain. TLS termination is handled by ACA's ingress -- the container listens on plaintext :8080.

### Cold Start Mitigation

ACA Consumption cold start for a 15MB Go container is 15-30 seconds. This is the infrastructure allocation overhead (Envoy sidecar, networking provisioning, image pull), not the binary startup time.

**Options in order of cost:**

1. **Accept + client retry ($0 extra):** MCP clients retry on connection failure. A 15-30s cold start on the first request after idle is acceptable for an MCP server where the client is an automated agent. Document the behavior. This is the Phase 1 recommendation.

2. **KEDA schedule rule (~$2-4/mo extra):** Keep 1 replica warm during business hours (e.g., 06:00-22:00 UTC weekdays), scale to 0 outside. This eliminates cold starts during active hours while still saving on overnight/weekend compute.

   ```bicep
   // Add to scale.rules
   {
     name: 'business-hours'
     custom: {
       type: 'cron'
       metadata: {
         timezone: 'UTC'
         start: '0 6 * * 1-5'
         end: '0 22 * * 1-5'
         desiredReplicas: '1'
       }
     }
   }
   ```

3. **min-replicas=1 (~$3-5/mo extra):** Always keep one replica warm. Eliminates cold starts entirely but burns through the free grant in ~8 days. Remaining ~22 days cost ~$3/mo in overage.

**Recommendation:** Start with option 1 (accept cold starts). Monitor cold start frequency via /health. If agents report connection failures, upgrade to option 2 (KEDA schedule). Option 3 is the fallback if cold starts are truly unacceptable.

## 6. Azure Managed Application (Marketplace)

### Backing Compute: App Service B1

App Service is the best backing for the managed app template:

- Most "Azure-native" container hosting option. Enterprise customers understand it.
- Always-on at $13.14/mo -- appropriate for a paid enterprise product. Zero cold start.
- Built-in custom domain + managed TLS certificates (free).
- Health check probes, auto-restart on crash.
- Deployment slots available on Standard tier ($54.75/mo) for zero-downtime updates.
- With Azure Table Storage, no persistent disk needed. Container is stateless.

### Bicep Template

The managed app template provisions:

1. **App Service Plan** (B1 Linux) -- compute
2. **Web App** (container from GHCR) -- cf-mcp process
3. **Storage Account** -- Azure Table Storage for session state
4. **Application Insights** -- telemetry
5. **Log Analytics Workspace** -- log aggregation (if customer lacks one)
6. **Managed Identity** -- accessing storage and App Insights without keys
7. **RBAC** -- Storage Table Data Contributor for managed identity

## 8. P0 Blockers

Every P0 blocker must be resolved before first public deploy.

### 8.1 Rate Limiting

**Problem:** No rate limiting exists. A single curl loop can exhaust session limits or spam campfires.

**Solution:** Application-level rate limiting in the HTTP handler.

- **Per-IP rate limit on `/mcp`:** 60 requests/minute using token bucket (`golang.org/x/time/rate`).
- **Per-session rate limit on `campfire_send`:** 30 messages/minute per session token.
- **Per-operator agent quota:** enforced at `campfire_init` time. Reject with clear error when operator hits agent cap for their tier.
- **Per-session storage quota:** 50MB equivalent in Azure Tables entities (~500 messages at 100KB each).
- **Per-session campfire creation limit:** 10 campfires per session.

Data structure: `sync.Map` of IP/token to `rate.Limiter`. Per-operator agent count tracked in `cfoperators` table (or derived from `cfsessions` partition scan on `operator_id`). Reaped by the session reaper.

File: `cmd/cf-mcp/ratelimit.go` (~80 LOC)

### 8.2 Graceful Shutdown

**Problem:** No SIGTERM handler. In-flight requests dropped on container restart or deploy. Azure Tables writes may be partially completed.

**Solution:** Signal handler with graceful drain.

```go
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)
defer stop()

go func() {
    <-ctx.Done()
    shutCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    srv.Shutdown(shutCtx)
}()
```

ACA sends SIGTERM with a 30-second grace period before SIGKILL. The 5-second drain is well within this window.

Files: modify `cmd/cf-mcp/main.go` (~15 LOC), add `CloseAll()` to `cmd/cf-mcp/session.go` (~10 LOC)

### 8.3 SSE Timeout Handling

**Problem:** `http.Server.ReadTimeout` (60s) kills long-lived SSE connections. App Service has a 230-second idle timeout at the load balancer.

**Solution:** Remove global `ReadTimeout`. Apply per-route timeouts. Add SSE keepalive.

```go
srv := &http.Server{
    Addr:              addr,
    Handler:           mux,
    ReadHeaderTimeout: 10 * time.Second,
    IdleTimeout:       120 * time.Second,
}
```

- `/mcp` POST: wrap with `http.TimeoutHandler(handler, 90*time.Second, "request timeout")`
- `/sse`: no timeout. Send keepalive comment (`: keepalive\n\n`) every 15 seconds. This keeps the connection alive through ACA's ingress timeout and App Service's 230-second idle timeout.
- `/campfire/*`: HTTP transport's own timeouts apply.

File: modify `cmd/cf-mcp/main.go` (~20 LOC)

### 8.4 Named Identity Collision

**Problem:** `campfire_init` with `name` in hosted mode resolves to a shared filesystem path. Two agents with `name='alice'` collide.

**Decision:** Prohibit named identities in hosted mode. Return a clear JSON-RPC error:

```json
{"code": -32602, "message": "named identities are not supported in hosted mode -- use session-scoped identity instead"}
```

File: modify `cmd/cf-mcp/main.go` (~5 LOC)

## 10. Cost Architecture

### Architectural Principles

Three optimizations dominate the cost model. In order of impact:

1. **SSE-first delivery.** Message delivery via in-memory SSE fan-out eliminates read transactions entirely. Without SSE, 1M agents polling at 10/min = ~432B reads/month = $1.55M/mo in table costs. With SSE, read transactions drop to near zero. This is a 100x+ cost reduction and the single most important optimization.

2. **In-memory cache + write-through.** Memory is the hot path. Azure Tables is the durable journal. 99%+ of reads hit the in-memory ring buffer. The only table reads occur on cold start (cache hydration) and cache miss (evicted campfire). This shifts cost from per-read transactions to per-container compute.

3. **Linear cost, shallow slope.** The cost curve is linear with agent count -- not sublinear. But the optimizations reduce the slope by 43-102x compared to naive polling. Honesty: a coordination protocol where every agent writes messages cannot have sublinear storage costs. The writes are the product. SSE and caching make the slope shallow enough that 1M agents costs ~$44K/mo (adversary model) to ~$500/mo (creative model), not $1.93M/mo (naive).

### What Does NOT Save Money

**Entity Group Transactions.** Azure bills per entity operation, not per HTTP call. A batch of 10 entities costs 10 transactions. Batching saves latency (fewer HTTP round-trips) but zero dollars in transaction billing. Any claim about batch cost savings is wrong.

### Cost Components

**Compute (ACA Consumption):**
- Active: $0.000024/vCPU-sec + $0.000003/GiB-sec
- Free grant: 180K vCPU-sec + 360K GiB-sec + 2M requests/month
- SSE connections keep containers alive 24/7 -- no scale-to-zero while any agent is connected
- Connection density: ~7,000 agents per 0.5 GiB container (practical limit after connection churn, TCP buffers, and application state)
- Per SSE connection: ~44KB (8KB goroutine stack + 32KB TCP buffer + 4KB application buffer)

**Azure Tables:**
- $0.036 per 10K transactions, $0.045/GB-month storage
- With SSE-first: writes dominate (reads near zero)
- Per message: ~2 transactions (message write + cursor update)

**Azure Blob Storage (Phase 3 alternative):**
- $0.006/10K write operations (6x cheaper than Tables)
- $0.001/10K read operations (36x cheaper than Tables)
- Append blobs: per-campfire, daily rotation (50K append limit per blob)

Implementation: each campfire's messages stored as an append blob. Per-campfire, daily rotation (50K appends per blob limit). Reads use the in-memory cache -- blob reads ($0.001/10K) only on cache miss or cold start hydration.

Trade-offs: append blobs lack the partition-scan query model of Tables (must read sequentially), max 195GB per blob, 50K append limit requires rotation logic. Suitable for the write-heavy journal pattern but requires more application-level complexity than Tables.

This is a Phase 3 optimization -- not needed until table write costs exceed compute costs.

## 12. Cold Start Strategy

Cold start has two phases: infrastructure allocation and cache hydration. Both matter.

### Phase A: Infrastructure Allocation (15-30 seconds)

ACA Consumption cold start for a 15MB Go container: 15-30 seconds. This is dominated by infrastructure allocation (Envoy sidecar, networking, image pull), not binary startup. The container is running but the cache is empty.

### Phase B: Cache Hydration (seconds to minutes at scale)

After the container starts, the in-memory cache must be populated from Azure Tables. Until hydration completes, every read is a cache miss that hits Tables at full transaction cost.

Hydration strategy: **lazy, not eager.** Campfire state loads on first access, not at startup. This means:
- Container is ready to serve immediately after Phase A
- First request to each campfire incurs a hydration penalty (partition scan on Tables, ~5-10ms)
- Subsequent requests to the same campfire hit the warm cache
- Full hydration happens organically as agents reconnect

| Scale | Active Campfires | Lazy Hydration Time | Table Reads During Hydration |
|-------|-----------------|--------------------|-----------------------------|
| 100 agents | ~20 campfires | Seconds (as agents reconnect) | ~20 partition scans |
| 1,000 agents | ~200 campfires | 1-2 minutes | ~200 partition scans |
| 10,000 agents | ~2,000 campfires | 2-5 minutes | ~2,000 partition scans |

At 10K+ agents per container, eager hydration (loading all state at startup) would take ~67 minutes for a full 4GB cache (4M entity reads at ~1,000 entities/sec). This is why lazy hydration is mandatory at scale -- the container cannot afford to be unresponsive for over an hour.

**Hydration cost:** During the hydration window, cache miss reads hit Tables. At 1,000 agents reconnecting over 2 minutes, that is ~200 partition scans + individual message reads = ~5K-10K table transactions (~$0.04). Negligible, but the latency penalty is real -- first reads per campfire are 5-10ms instead of sub-millisecond.

### Assessment

15-30 seconds for infrastructure + lazy hydration on first access is tolerable for an MCP server where the client is an automated agent that retries. MCP clients (Claude Code, Cursor, etc.) handle connection failures with automatic retry.

### Recommendation

**Phase 1: Accept cold starts + lazy hydration.** Document the behavior. MCP clients retry automatically. The hosted service is free -- users accept the trade-off.

**Phase 1.5 (if cold starts cause adoption friction):** Add a KEDA cron scale rule to keep one replica warm during business hours. Cost: ~$2-4/mo extra. This eliminates Phase A but cache state is preserved (no hydration needed if the container stays warm).

**Phase 2 (managed app):** App Service B1 is always-on. No infrastructure cold start. Cache stays warm across requests. Cold start is not an issue for paid customers.

The key insight: with SSE-first architecture, cold start is rare. Once any agent holds an SSE connection, the container stays warm and the cache stays populated. Cold start only happens when literally zero agents are connected -- and at that point, the lazy hydration cost is proportional to the first few reconnecting agents, not the total agent count.

## 13. SSE and Scale-to-Zero

### SSE-First Is an Architectural Choice, Not a Limitation

SSE is the primary delivery mechanism, not a side effect. The entire cost architecture depends on SSE fan-out eliminating table read transactions (see Section 10). Polling is supported as a fallback for clients that cannot hold SSE connections, but the system is designed and optimized for SSE-connected agents.

This means scale-to-zero is deliberately sacrificed for cost efficiency at scale. The trade-off is explicit:

| Scenario | Scale-to-Zero? | ACA Monthly Cost |
|---|---|---|
| No agents connected | Yes | $0 |
| Agents connect, do work, disconnect | Yes (between sessions) | ~$0.01-1 |
| 1+ agent with persistent SSE | No -- container stays running | ~$19/mo |
| Many agents, all with SSE | No | ~$19/mo (one container handles ~7K agents) |

The cost floor with any persistent SSE client is ~$19/mo (0.25 vCPU + 0.5 GiB, 24/7). The "scale-to-zero = $0" story only holds when literally zero agents are connected.

**Why this is the right trade-off:** Without SSE, 1M agents polling at 10/min = ~432B table reads/month = $1.55M/mo. With SSE, the same 1M agents cost ~$9,300/mo in compute (150 containers) + ~$2,600/mo in table writes = ~$12K/mo. SSE is a 130x cost reduction at scale.

### Mitigation Path

**Current (HTTP+SSE):** Accept the ~$19/mo floor per container. This is cheap and the right architecture.

**Future (Streamable HTTP):** The MCP specification's Streamable HTTP transport eliminates the persistent SSE requirement. The server sends responses as regular HTTP responses. SSE is optional for server-initiated notifications. When MCP clients adopt Streamable HTTP, true scale-to-zero becomes possible for low-traffic deployments. However, SSE remains optimal for cost at scale -- Streamable HTTP would reintroduce polling costs.

Streamable HTTP is Phase 3 work. Its value is for small deployments where scale-to-zero matters more than per-read cost.

One image serves both ACA (hosted) and App Service (managed app). The binary is environment-agnostic. Configuration comes from flags and environment variables.

## 15. Session Model

### Ephemerality as Feature

Default session TTL is 48 hours of inactivity. Expired sessions are deleted from Azure Tables. `campfire_export` is the persistence valve.

**Rationale:**
- Most MCP-connected agents live minutes to hours. 48 hours covers weekend gaps.
- With Azure Tables, expired session cleanup is a table query + per-entity deletes. No disk management.
- The hosted service documents: "Your session is ephemeral. Export before you leave."

### Session Reaper on Azure Tables

The session reaper runs as a goroutine in the cf-mcp process. Every 5 minutes, it queries `cfsessions` for entities where `last_active < now() - 48h` and deletes them along with their associated data in `cfmemberships`, `cfcursors`, and `cfthresholdshares`.

Because Azure Tables has no cross-table transactions, orphaned entities may exist briefly if the reaper crashes mid-cleanup. This is acceptable -- orphaned entities are harmless (unreachable without a session token) and will be cleaned up on the next reaper cycle.
